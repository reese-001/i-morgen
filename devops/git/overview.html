    • Overview

    • Git has three main types of entities that it tracks
      • Files
        • Git tracks changes made to individual files in your repository
        • Every time you make a change to a file, Git records that change as a "commit"
      • Commits
        • A commit is a snapshot of your repository at a specific point in time
        • Every commit contains a set of changes to the files in your repository, along with a message that describes the changes
        • Commits are identified by a unique hash, and they form a linear history of your repository
      • Branches
        • A branch is a pointer to a specific commit in your repository
        • By default, Git has a single branch named "master", but you can create additional branches to track different versions of your code
        • Branches allow you to work on multiple features or bug fixes simultaneously, without interfering with each other

    Best Practices

    • Avoid checking in code to the main branch, use feature branches instead
      • Why?
        • Branches allow code reviews via pull requests
        • Branches help avoid breaking the code
      • How? (specific to CodeCommit)
        • Limit pushes and merges to branches via IAM deny policies
          • Allows you to create groups (ie. senior vs junior devs) and allow seniors to initiate pull requests
    • Keep commits small and focused
      • Why?
        • Makes it easier to track changes and revert changes if necessary
      • How?
        • Each commit should ideally have a single focus and should contain only related changes
    • Write clear and concise commit messages
      • Why?
        • Allow reviewers and others to understand what is going on
      • How?
        • The commit message should summarize the changes made in the commit and should be written in a clear and concise manner

    Storing Data

    • Upon commit, Git stores a snapshot of the entire file, not a diff
      • This allows Git to provide a clearer view of your code changes over time
      • Makes it easier to revert to a previous version in your codebase
    • Git is built for speed, not for storage optimzation
      • To see this, review the files in the .git/objects directory
      • There are no delta files
    • Pack operation (git pack) is used to create a single compressed file that contains multiple Git objects, such as commits, trees, blobs
      • Output is called a pack file
      • Typically performed automatically by Git (along with other routine items such as git gc)
      • You can perform manually however
      • Things to keep in mind
        • Pack files are read only
        • When Git needs to access a Git object that is stored in a pack file, it will first look for an index file that maps the object's SHA-1 hash to its location in the pack file. If the index file does not exist, Git will create one automatically
        • If a Git object is not present in the pack file, Git will retrieve it from its original location in the repository

    Origin

    • ** UNCONFIRMED **
    • The name "git" is a reference to the British slang term "git", which is a derogatory term for a foolish or unpleasant person
    • The name was chosen by Linus Torvalds, the creator of Git, as a way to poke fun at himself and the challenges he had faced while developing the tool
    • According to Torvalds, he initially intended to call the tool "The Stupid Content Tracker" as a nod to the fact that he had been frustrated with existing version control tools
    • However, he decided to name the tool "git" instead, as he felt that it was a more memorable name
