Brief History of Containers

  • 1970's: chroot became available which allowed you to change the root directory for a process and its children to a new directory, effectively creating a new, isolated environment
  • 2001-2004: Solaris introduced Zones and BSD came up with jails which were similar to what we can Linux Containers/OS virtualization
    • In this configuration, you had to run the virtual environment with the same OS as the host
  • 2005: OpenVZ was created which allowed all containers to share a single kernel
    • Due to lack of orchestration and complexity of software, it did not gain a large following
  • 2003-2006: Google's container orchestration tool called Borg was being developed See DevOps History 2008-2009
    • Contributed cgroups to the Linux kernel to allow for resource restrictions cpu, memory, etc for containers
    • Additionally Google contributed namespace code to the kernel
    • In 2008 these two contributions lead to the release of LXC (Linux Containers)
  • In 2009 a company called dotCloud released a web based application that utilized LXC
    • This product lead to the creation of Docker in 2012
    • See DevOps History 2010-2015
  • In 2014 Google release Kubernetes and Docker releases Swarm
    • See DevOps History 2010-2015
    • AWS released EKS and AKS
    • Kubernetes has an interface (CLI) that allows for multiple other container platforms to be operated
  • In about the same time period, Marathon was also released by Apache

Docker - What's Under the Hood?

  • Allows the creation of a containerized environment without requiring a virtual machine
  • This occurs via the following technologies
    • Namespaces
      • What appears as containers is actually just Linux Namespaces
      • Functions like network, root file system, users, etc are virtualized
      • Nothing else is however
    • Cgroups
      • Offer resource isolation
    • Layered file system
      • When you create a Docker image, Docker uses a layered file system to build the image
      • Each layer in the file system represents a change to the file system made by the Dockerfile instruction that created it

Kubernetes - Why Do We Need It?

  • Kubernetes offers two things
    • Packaging format
    • Ability to run that package in a contained environment
  • Docker Compose allows you to run multiple containers on a single machine
  • However there are a few problems with this
    • Scheduling: how do you decide what container runs where?
    • Network: How do you network the containers together?
  • In addition there are the typical non functional requirements (NFR's)
    • Availability
    • Scalability
    • Security
  • Kubernetes solves these problems by taking these servers and creating one logical entity
    • No matter how many workloads we have, you submit it to Kubernetes and it schedules it for you
      • Workload refers to a set of instructions that define how a container or group of containers should be deployed and managed on a Kubernetes cluster.
      • Workloads are used to define the desired state of the application and Kubernetes is responsible for maintaining that state
    • Facilitates the creation of the networking for you
      • Kubernetes does not do this itself, but has numerous plugins to assist with this
    • Takes care of the NFR's mentioned above
      • You get HA out of the box
      • Allows for a variety of scaling schemes
      • Provides many tools for a secure deployment and operation

Walk Through: Docker

  • Goal: Clone Git repo, create and load Docker image
    • Locate a repository that is designed to work with Docker - for example "https://github.com/lfs161/facebooc.git"
    • Copy the code and execute: git clone https://github.com/lfs161/facebooc.git
    • Build Docker image: docker image build -t facebooc:v1 .
    • Run Docker image: docker run -idt --name fb -p 16000:16000 facebooc:v1
    • Navigate to local ip and correct port in browser

Is Docker Dead?

  • In 2015 the Linux Foundation created the Open Container Standards (OCI)
    • Among other things, it created a standardized format for container images that can be used by any container runtime that supports the OCI standards
    • This led to the creation of a runtime known as runc
      • Responsible for creating namespaces, control groups, etc
    • Docker created its own runtime based on runc that is called containerd
      • Pulls image, pulls registry, runs as a daemon
  • Docker was primary in the early years, however now Kubernetes is primary
    • The world of containers revolves around Kubernetes
    • The Kubernetes/Docker interface is called DockerShim
    • However for everything else, Kubernetes uses CRI (Container Runtime Interface)
    • Kubernetes has indicated they will stop supporting DockerShim due to stability issues
  • This means that Docker will not be supported in production environments that require orchestation
    • However Docker will still be used for dev and personal use due to simplicity
    • Docker images should be compatible with CRI-O and ContainerD runtime

Emerging Tech

  • podman
    • Podman is a daemonless container engine for developing, managing, and running OCI Containers on your Linux System
    • Containers can either be run as root or in rootless mode
  • buildah
    • Facilitates building OCI container images
  • kaniko
    • Used to build container images from a Dockerfile, inside a container or Kubernetes cluster
  • skopeo
    • Helps manage your images and work with multiple registries
