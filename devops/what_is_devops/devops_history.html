Brief History of DevOps


Contents
  • 2000 - 2008
  • 2008 - 2009
  • 2010 - 2015
  • 2016 - 2018
  • Current State

2000 - 2008

  • Deployments
    • Deployments typically occured in intervals ranging from weekly to every few months
    • These deployments often occured during maintenance windows to allow for downtime - typically outside of business hours
    • They were generally much more manual then modern processes
      • Often involved copying files from dev servers to production environments
      • Deployments often required downtime
      • Manual testing was the norm
        • Hudson (eventually Jenkins in 2010) was developed in 2004 and was starting to get traction
        • CruiseControl was possibly the first CI tool and was created in 2001 by Martin Fowler and Kent Beck
      • Code repos were centralized and managed by a small group of developers
      • Configuration management software such as Puppet and Chef were just coming on scene
      • Monitoring software existed, but not in its modern form
        • Nagios was released in 1999 and was in widespread use during this time frame
        • Allows for metric gathering and alerts, but the alerts often triggered too often due to limitations in the product
        • PagerDuty released in 2009
  • Cloud Computing
    • In general, cloud computing was still in its infancy at this point
      • There was not widespread agreement that it was generally advantageous due to security and vendor lock in concerns
      • AWS EC2 and S3 became available on 2006
      • Rackspace launched in 2006
    • Companies that used AWS in the last 2000's include
      • Foursquare
      • DropBox
      • SmugMug

2008 - 2009

  • Configuration Management
    • Tools like Puppet (2005) created the ability to centrally drive configuration management
      • Puppet is an open-source systems management tool created for centralizing and automating configuration management
      • It includes its own declarative language to describe system configuration
    • Introduced the concept of IaC, which in this case allowed you to publish your code to a centralized server and all your systems would pull the changes
    • When Google started to use Puppet, it created increased interest
      • Even though they primarily used it for laptops and desktops
  • Containers
    • Containers became mainstream about 2013
    • Google became aggressive in containerization
      • Contributed code to Linux kernal that eventually allowed the creation of Linux Containers (LXC)
      • Created an orchestrator called "Borg" in 2004 that became Kubernetes
  • Devops coined for the first time in 2009
    • Convergence of multiple factors
      • Agile infrastructure and operations: how infra-file are you? (2008) link
      • Agile Systems Administrator Group formed by Andrew Clay Shafer and Patrick Debois in 2008
      • Devops Days started by Patrick Debois in 2009
  • Configuration Management
    • Chef (2009)
      • As per the Git writeup, Chef is a " powerful automation platform that transforms infrastructure into code automating how infrastructure is configured, deployed and managed across any environment, at any scale"
      • Uses "cookbooks" and "receipts" to translate system admin tasks into reusable definitions
      • Runs on a mix of platforms to include on prem and cloud environments
    • Puppet and Chef were the two primary configuration tools available at this time, and would have been well known by anyone doing this kind of work

2010-2015

  • By this time the word "DevOps" was in use, but it was not fully defined and there was a wide variety of competing technologies available
  • Introduction of Jenkins
    • In 2011, Hudson and Jenkins split
    • There were many options for CI, but Jenkins became the "boss"
    • Other options included (and still do include)
      • Travis
      • CircleCI
  • Also in 2011/2012, Michael Dehaan (originally from Puppet) developed Ansible
    • "Ansible is a radically simple IT automation engine that automates cloud provisioning, configuration management, application deployment, intra-service orchestration, and many other IT needs" source
    • Incorporated in 2013 and sold to RedHat in 2015
  • The appearance of Docker
    • Linux Containers (LXC), which was discussed above, initiated interest in containerization
    • A company called dotCloud was created to develop a web-based collaborative IDE environment for developers
    • However by 2012, possibly due to funding issue, dotCloud took a different approach and decided to open source their technologies
    • This became Docker and it became extremely popular
  • Kubernetes
    • Running Docker at scale and in production was the problem
    • In 2014 Google decided to open source their solution that they called Kubernetes
      • As mentioned above, Borg was the internal Google solution to orchestrating containers
      • Legend is that Borg needed a friendlier face, so they named the project "Seven of Nine" from STNG. The seven spokes on the wheel came from that
  • Kerbernetes was not the only option
    • Docker released their orchestation technology called Swarm
      • Swarm is a simpler and more lightweight platform than Kubernetes or Mesos, and is designed for smaller-scale container deployments
    • Apache released their tool called Mesos
      • Mesos is designed to manage multiple resources in a distributed environment, including CPU, memory, and storage,
      • Marathon provides a framework for deploying and scaling containerized applications on top of Mesos
      • Mesos is a more general-purpose platform that can also manage other types of workloads beyond containers, such as big data processing and machine learning
  • Pheonix Project
    • Fictional novel about a newly minted IT executive thrown into a seemingly impossible situation
    • He is guided by a mentor who is familar with the disciplines of lean manufacturing

2016-2018

  • Kubernetes emerges as gold standard
    • 2016 saw introduction of scaling, presets, windows support and tools such as Helm, minikube, kubeadm, kOps
    • By 2017 enterprises started endorsing Kubernetes by either migrating to it or releasing products around it
      • AWS and Azure release K8 based products
  • Ansible becomes standard
    • Docker and Kubernetes has eliminated a large portion of the application configurations
    • We are left with the configuration of the underlying systems, patching and compliance
    • Ansible fills the requirement

Current State

  • There are many ways to define devops - here is another: the five pillars of devops
    • Cloud
      • AWS
      • Azure
      • Google Cloud
    • Containers
      • Kubernetes
        • Supports other run times besides Docker
        • Watch this: CRI-O
      • Docker
        • Consider the connection to shipping containers and the standards that allow transportation regardless of vehicle (train, ship, truck)
    • IaaC
      • Ansible
      • Terraform
      • There are many other applications that serve this purpose as well
    • CI/CD
      • Jenkins
      • Spinnaker extends the features of the pipeline and setting up CD
      • Everything else
    • Observability
      • This is more important in a world of distributed microservices
      • Logs
        • ELK Stack
        • ElasticSearch
        • Kibana
        • Logstash
      • Metrics
        • Prometheus (more like a time series database)
        • Grafana (can be used to visualize Prometheus data)
      • Traces
        • Jaeger
        • Zipkin
      • Tools such as Prometheus allow for multi-dimensional analysis
      • Observability is not just about setting the tools up however, its about a culture
