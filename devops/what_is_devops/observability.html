Why Bother

  • Observability is crucial in application design for several reasons, as it directly impacts the reliability, performance, and maintainability of an application.
  • Key reasons
    • Monitoring and troubleshooting
    • Performance optimization
    • Reducing mean time to resolution (MTTR)
    • Proactive issue detection
    • Enhanced collaboration
    • Continuous improvement
  • Can be divided into three components - Refer to the Three Pillars of Observability link
    • Event logging is a critical aspect of observability that involves recording and storing information about various events that occur within a system
    • Dashboards are a key component for visualizing and understanding the data generated from an observable system
    • Tracing provides detailed information about individual requests or transactions as they pass through various components of a distributed system, offering insights into the system's behavior and performance at a granular level

Metrics, Logs and Traces: What To Observe and Why

  • Metrics
    • Data that represents the health of the system
    • Generally you would store this data in a time series database - see link
    • Common tools
      • Prometheus
        • This is often tied with Grafana
  • Logs
    • Due to the number of servers/VM's/containers, it is not feasible to search for logs on individual machines
    • It is important to aggregate your logs from the individual servers and place them into a central location
    • One popular tool set to accomplish this is the ELK stack
      • ElasticSearch: Search and analytics engine designed for handling large volumes of data
      • Logstash: A data processing pipeline that ingests, processes, and enriches log data from various sources before forwarding it to Elasticsearch for indexing and storage
      • Kibana: A web-based visualization and exploration tool for Elasticsearch data
    • Other tools
      • Splunk: A powerful commercial log management and analysis platform that offers real-time search, monitoring, and analytics capabilities for various data types, including logs, metrics, and events
      • Graylog: An open-source log management platform that centralizes, processes, and analyzes log data. Graylog uses Elasticsearch for search and storage, and its web interface allows users to create custom dashboards and visualizations
      • Sumo Logic: An open-source log management platform that centralizes, processes, and analyzes log data. Graylog uses Elasticsearch for search and storage, and its web interface allows users to create custom dashboards and visualizations
      • Papertrail: A cloud-based log management solution that aggregates logs from various sources, allowing users to search, monitor, and set up alerts. Papertrail offers a user-friendly interface and integrates with other tools like PagerDuty, Slack, and custom webhooks
      • Loggly: A cloud-based log management and analysis service that offers features like log aggregation, search, visualization, and alerting. Loggly is designed for scalability and ease of use, providing insights into application performance, infrastructure monitoring, and security
      • Fluentd: An open-source data collector for unified logging layers. Fluentd helps in log aggregation and forwarding, providing a flexible plugin system that allows users to transform, enrich, and output log data to various destinations like Elasticsearch, S3, or other storage and analysis systems
  • Traces
    • Tracing provides detailed information about individual requests or transactions as they pass through various components of a distributed system
    • Assists with the following
      • End-to-end visibility
      • Root cause analysis
      • Correlation with other observability data
      • Trace-driven dashboards
      • Monitoring service-level objectives (SLOs)
    • Tool set
      • See DevOps History link
      • Jaeger: open-source, end-to-end distributed tracing system developed by Uber. It supports the OpenTracing API and provides a rich set of features, including context propagation, span creation, and trace visualization
      • Zipkin: open-source distributed tracing system inspired by Google's Dapper paper. Originally developed by Twitter, Zipkin provides the ability to trace requests across multiple services and visualize latency data, helping teams understand their application's behavior and identify performance bottlenecks
      • OpenTelemetry: set of APIs, libraries, and agents that provide observability for applications by capturing and exporting telemetry data, including traces, metrics, and logs. OpenTelemetry is a CNCF project that aims to provide a unified observability framework, merging the efforts of OpenTracing and OpenCensus
      • Datadog APM: commercial application performance monitoring (APM) and tracing solution offered by Datadog. It provides end-to-end request tracing, detailed flame graphs, and powerful analytics features to help teams understand the performance of their distributed applications and services
      • New Relic APM: Another commercial APM solution, New Relic APM offers distributed tracing capabilities to visualize and analyze request traces across multiple services. It also provides features like error analytics, custom metrics, and integrations with other observability tools
      • Lightstep: commercial observability platform that provides distributed tracing and performance analysis for large-scale, complex systems. Lightstep uses a unique approach called "satellite architecture" for efficient data collection and analysis, allowing teams to gain insights into their applications' behavior in real-time

SLAs, SLOs, SLIs and Observability

  • SLI
    • Service Level Indicators (SLIs) are quantitative measures used to evaluate the performance and quality of a service
    • SLIs are the metrics that are tracked to determine whether a service is meeting its defined Service Level Objectives (SLOs)
    • They provide a way to measure specific aspects of the service, such as availability, latency, throughput, or error rates
    • Common SLI's include
      • Availability
        • Latencys
      • Throughput
      • Error Rate
    • Example: 95% latency of homepage requests in the last 5 min < 300 ms
  • SLO
    • SLOs are the specific, measurable performance goals that a service provider aims to achieve for a particular service
    • SLOs are typically expressed as target values or ranges for key performance metrics, such as availability, latency, or error rates
    • Example: The SLI will be positive 99.9% of the trailing year
  • SLA
    • Service Level Agreements (SLAs) are formal, documented agreements between a service provider and a customer that define the expectations, performance metrics, and service quality standards the provider is expected to meet
    • Commonly includes
      • Scope of services
      • Performance metrics
      • Service level objectives (SLOs)
      • Monitoring and reporting
      • Escalation and remediation
      • Review and revision
  • Error Budgets
    • Represents the acceptable level of unreliability or risk a service is allowed to have while still meeting its performance goals
    • It is derived from the Service Level Objective (SLO) and serves as a buffer to balance reliability with the need for innovation and rapid development
    • Example
      • An error budget is calculated by subtracting the target reliability (SLO) from 100%
      • If a service has an SLO of 99.9% availability, the error budget would be 0.1% (100% - 99.9%)
      • This means that over a specific period, the service is allowed to have 0.1% downtime or unavailability without violating the SLO
    • What happens when the error budget deviates in any direction?
      • When the error budget is exhausted or near exhaustion, it is an indication that the focus should shift towards improving the service's reliability and stability
      • If there is a significant amount of unused error budget, it could be a sign that the service is overly conservative and may benefit from faster innovation or deployment of new features
