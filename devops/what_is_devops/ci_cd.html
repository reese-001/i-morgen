Continuous Integration/Continuous Delivery (CI/CD)


Note: See aws/CI/CD page here

Contents
  • Why Bother?
  • What is CI/CD
  • CI/CD Principles and Practices
  • Understanding Release Strategies

Why Bother?

  • One of the most important features of your application is reliability
    • What does it mean to be reliable?
      • The application must be available when your customers want it
      • It must be scalable
      • Software defects must be controlled
    • To minimize defects, feedback must be given timely
    • The ideal solution then is to initiate code testing/scans on merge requests and this is what CI provides
  • CD automates the path to deployment

What Is Continuous Integration

  • "Simple Testing Can Prevent Most Critical Failures" link
  • Process of ensuring that every change that gets integrated into the main line of the repository goes through enough checks that we are confident about its reliability
  • Common workflow and integration scenario
    • Work in teams but share the code base
    • Everyone checks out the code from a common repo
    • Maintain a mainline version of the code, typically a master or main branch
    • Teams create their respective branches of development
    • To support agile practices, work on smaller batches of changes, and integrated often by merging the code to master/main
  • When do we run the tests?
    • We run the tests whenever we integrate to the main line of the code
    • Simply, whenever the code integrates, run the tests
    • Note: that this is a simplified answer and in cases like security scans testing should be done more often
  • What do we test
    • Note: Not all of these are required. However build and unit tests should always be run
    • Build tests
    • Unit tests
    • Static analysis (ie sonarqube)
    • Integration tests
    • Acceptance tests
    • Load tests
    • Compliance tests
  • How do we do this
    • Desired features of a CI tool
      • Ability to create jobs to run build, tests, packaging, etc by integrating with various tools
      • Connect those jobs to create a sequence of execution pipelines
      • Run those pipelines in an automated way
      • Ability to trigger these pipelines based on external events (ie pushing code to a git repo)
    • Tools
      • Jenkins
      • Go
      • Bamboo
      • Web hosted platforms such as
        • CircleCI
        • TravisCI

Continuous Deployment and Continuous Delivery

  • Extends CI to potentially take every change directly to production
  • You gain confidence to do that because your testing was comprehensive and complete
  • There are two flavors here
    • Continuous Deployment
      • Code changes are automatically prepared for a release to production, but not deployed until they are manually approved by a human
    • Continuous Delivery
      • Code changes are automatically prepared and deployed to production as soon as they pass automated testing
      • This is relevant to a large number of organizations
      • Provides control and a safer approach to protecting production
      • Allows for manual and exploratory testing
      • Could accommodate hybrid deployments (e.g. update some schemes by hand then auto deploy)
      • May make release managers happy

CI/CD Principles

  • Prerequisites
    • Agile
      • CI/CD requires you work in smaller batches and creating new features faster
      • This is what Agile helps you with
    • Build action needs to meet the following criteria
      • Quick: You cant achieve continuous feedback if the build process takes hours (or more)
      • Automated: Maven (Java), NPM (Node.js), etc
    • Tests
      • Should be automated
      • Refer to the Test Pyramid
        • Testing pyramid is a framework for designing and prioritizing different types of tests in a CI/CD pipeline
        • The goal of the testing pyramid is to focus testing efforts on the most important and impactful areas of the codebase, while maintaining a balance between test coverage, speed, and reliability
    • Deployments
      • If your goal is to deliver the product, it is important to consider automated deployments
  • Practices
    • Revision control
      • What should we control?
        • Application code
        • Build logic (pox.xml, package.json, makefiles)
        • Configurations
        • CI Pipelines (e.g. Jenkinsfile)
        • Artifact build and packaging logic (e.g. Dockerfiles)
        • Infrastructure as Code (IaC) (e.g. Ansible)
        • Deployment logic (e.g. Kubernetes, deployment files)
    • Development Workflow
      • Defines how fast you can deliver and in which way are you delivering the product
      • Simplest workflow is a trunk based model
      • Alternatively you can use GitFlow
  • Achieving CI/CD
    • Foundation of CI/CD is the automated pipeline
    • Achieving Continuous Delivery
      • Setup revision control system
      • Define development workflow
      • Setup CI Platform
      • Create CI Pipelines - Built/Test
      • Build and manage artifacts - packaging jobs
      • Setup deployment tools and create deployment pipelines
