    • Overview

    • DevSecOps is a practice that integrates security considerations into the software development life cycle (SDLC) and the operations of an organization
    • It aims to bridge the gap between development, security, and operations teams, by promoting collaboration, communication, and automation of security activities throughout the entire SDLC
    • Expands upon the principles of Devops by shifting security considerations and testing to the left with the goal of identifying vulnerabilities early in the SDLC cycle - and prior to deployment
    • Key focus areas
      • Automating Security
        • Pipeline based security
          • Pre commit hooks
            • Scripts that are executed prior to a GIT commit
            • Used to enforce certain rules or checks on the code being committed
            • In the context of IT application security, pre-commit hooks can be used to
              • Validate code for security vulnerabilities
              • Format code to meet security standards
              • Ensure the code is in compliance
          • Source composition analysis (SCA)
            • Analyzes the components and dependencies of an appplication to identify known vulnerabilities and potential security risks
            • Typically involves scanning source code and metadata to identify included packages and compares them to databases of known vulnerabilities
            • Helps identify issues early in the process
            • Products include
              • Sonatype Nexus
                • Through component analysis, it provides insights into the security, quality, and licensing of components
              • Black Duck Hub
                • Provides component inventory management, vulnerability management, policy management and pipeline integration
              • Snyk
                • Provides vulnerability management, continuous monitoring, community vulnerability database and pipeline integration
              • Trufflehog
                • Can scan a Git repository and find secrets (e.g., API keys, passwords, etc.) that were accidentally committed to the repository
          • Static App Security Testing (SAST) (Dependency-Check, Requires.IO, Retire.js)
            • Executes white box testing post check in
            • Analyzes source code of an application for potential vulnerabilities without executing
            • Application does not need to be complete or to be functional.
            • Looks for patterns and anomalies that indicate vulnerabilities such as
              • Missing input validations
              • Hard coded passwords
              • Buffer overflows
            • Products include
              • SonarQube
              • Checkmarx
              • Veracode Static Analysis
              • Snyk
          • Dynamic Application Scanning Tools (DAST)
            • Similar to SAST, except it anaylzes the executing of an application and has no access to source code
            • Occurs post code build and often takes place in an isolated code sandbox
            • Does this by sending various inputs and observing output
            • What does DAST do?
              • Correct logging
              • Access controls
              • Specific vulnerabilities
                • Cross site scripting (XSS) link
                • Cross site request forgery (CSRF) link
                • SQL Injection attacks link
            • Products include
              • Acunetix
              • AppScan
              • Checkmarx
          • Security in Production
            • Automated patching
            • Configuration management

    Importance

    • The risk profile of your project extends beyond the code you write
      • Typically 10% to 20% of code is custom code
      • The remainder is open source code and libraries, which in turn often import other libraries
      • Containers and IaC are other large sources of vulnerabilities
    • Security teams provide expertise and tooling to increase developer autonomy while still providing a level of oversight
    • Six benefits Sec brings to DevOps
      • Faster delivery
        • Speed of delivery is improved when security is integrated into the pipeline
        • Defects and vulnerabilities are fixed before going into production
      • Improved security posture
        • A shared responsibility model ensures security is tightly integrated
      • Reduced costs
        • Identifying vulnerabilities before going to production reduces overall costs
      • Enhancing the value of devops
        • Improving the overall security posture as a result of collaboration between development and security
      • Improving security integration and pace
        • Costs are reduced by eliminating the need to retrofit security controls post development
      • Enabling greater overall business success
        • Greater trust in the ability to bring new technologies online increases the speed and efficiency of doing business

    General Guidance

    • Minimize footprint
      • Dont give hackers more tools to expand their exploits
      • Choose the slimest container
    • Layer housekeeping
      • Understand how layers work at build and run time
    • Build strategies
      • Multi-stage, repeatable builds, standardized labeling, alternative tools
    • Secure supply chain
      • Understand your chain of custody for libraries
    • Dont run as root
    • Do not use privledged containers
    • Drop capabilities you dont need
    • Read only root filesystem
    • Deploy from known sources
