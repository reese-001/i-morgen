What is it?

  • CodeBuild is a fully managed build service that compiles source code, runs tests, and produces software packages that are ready to deploy.
  • CodeBuild eliminates the need to provision, manage, and scale your own build servers.
  • It scales continuously and processes multiple builds concurrently, so your builds are not left waiting in a queue.
  • You can get started quickly by using prepackaged build environments, or you can create custom build environments that use your own build tools.
  • You can also fully automate the build process by using the AWS SDKs to integrate CodeBuild with your source control repository and deployment pipeline.

Workflow

  • Retreive the source code from Github, CodeCommit, CodePipeline, S3, etc
  • Build instructions can be defined in code ( buildspec.yml )
  • Output logs to S3 and CloudWatch logs
  • Monitored using CloudWatch metrics and get the statistics from them
  • Use CloudWatch Events to detect failed builds and trigger notifications
  • Use CloudWatch Alarms to notify if there are too many failures
  • Can send notifications to SNS using CodeBuild triggers

Build Environments

  • Docker images are managed by CodeBuild
  • Check the image to see what is preinstalled
  • Option: "Enable this flag if you want to build Docker images or want your builds to get elevated privileges"
    • CodeBuild launches the build environment with elevated privileges, allowing the build to perform tasks that require root-level access or access to Docker
    • This includes running Docker containers, building Docker images, and accessing resources outside of the build environment
    • Note that enabling this option can also increase the security risks associated with running builds, as the build environment is given higher-level permissions

Timeouts

  • Refers to the maximum amount of time that a build is allowed to run before it is stopped.
  • There are two types of timeouts in CodeBuild:
    1. Build timeout
      • Applies to the entire build process, including the time it takes to download the source code, execute the build commands, and upload the build artifacts.
      • You can set the build timeout to any value between 5 minutes and 1 day.
      • If the build takes longer than the specified timeout, it will be stopped and marked as a failure.
    2. Individual phase timeout
      • This timeout applies to each phase of the build process separately
      • There are three phases in a CodeBuild build
        1. Pre-Build: This phase occurs before the build starts, and is typically used to download the source code and install any dependencies.
        2. Build: This phase is where the build commands are executed.
        3. Post-build: This phase occurs after the build has completed, and is typically used to upload the build artifacts to Amazon S3 or another storage location.
        4. If any phase takes longer than the specified timeout, it will be stopped and the build will be marked as a failure.

Use Cases

  • Build and test code
  • Compile code
  • Run unit tests
  • Package artifacts
  • Deploy artifacts
  • Generate out static pages from a JAMstack
  • A website built with Gastby needs to render out static pages and delivery for a S3 static website hosting
    • Trigger with AWS Console
    • CodeBuild will pull from its source (CodeCommit)
    • CodeBiuld will render out static pages to S3
  • Developer needs to ensure their code passes all tests before being allowed to make a Pull Request
    • Push code to a feature branch
    • Github will trigger a web hook that triggers a lambda function using AWS SDK
    • CodeBuild pulls down project and runs build commands
    • Test coverage results passed to another lambda function and back to GitHub

Buildspec.yml

  • Refer to link in links section
  • Provides the build instructions and needs to be at the root of your project folder
  • Components
    • version: 0.2 recommended
    • Affects default shell in the build environment
      • v .1 runs each build command in a seperate instance
      • v .2 runs all build commands in the same instance
    • Environmental variables and parameter store key/value pairs
    • Phases: commands the run in each phase of the build
      • install: installing packages or dependencies that the build requires
      • pre_build: commands that run before builing
      • build: commands that run during build
      • post-build: commands that run after build
        • Can contain a 'finally' block in case something goes wrong and you need to clean up
    • Artifacts
      • What is kept after the build is done
      • Anything that is created in the container during the build process will be destroyed when the container is removed
      • If you need those artificts, identify them here and they can be preserved in S3
  • Example 1 
    • version: 0.2
phases: 
    install:
        runtime-versions:
            nodejs: 10
        commands:
            - echo "installing something"
    pre_build:
        commands: 
            - echo "we are in the pre build phase"
    build:
        commands:
            - echo "we are in the build block"
            - echo "we will run some tests"
            - grep -Fq "Congratulations" index.html
    post_build:
        commands:
            - echo "we are in the post build phase"
  • Example 2 (from this web site) 
    • version: 0.2
phases:
  install:
    runtime-versions:
      python: 3.9
    commands:
      - pip install -r requirements.txt
  build:
    commands:
      - python menu_script.py
artifacts:
  files:
    - 'templates/menu.html'

Logging

  • Select where you want to send the logs
  • Options include
    • CloudWatch
    • S3
  • If you do not set this up, you will lose the logs that were generated in the Docker container

Creating a Code Build Project

  • Assume a starting point of selecting "Create Build Project"
    • Not all options will be called out
  • Source
    • Options include (Feb 2023)
      • S3
      • CodeCommit
      • GitHub/Enterprise
      • BitBucket
    • Reference type
      • Branch
        • How to test ongoing code
      • Git Tag
        • For example, did you tag a commit to read "V1"? If so, you can select it here
      • Commit ID
        • Do you want to test a specific version of the repo
      • Additional configuration
        • This is more advanced and not yet reviewed
  • Environment
    • Environmental Image
      • Managed vs Custom
        • Do you need some specific software installed on your image to perform build and test
    • Additional Configuration
      • Note that the range of timeouts is from 5 min -> 8 hrs
        • This contrasts with Lambda with a timeout of 15 min which limits the ability to perform build and test ops
