    • What is it?

    • Fully managed solution for collecting, processing and analyzing streaming data in the cloud - alternative to Apache Kafka
    • Create data processing applications known as Kinesis Data Stream applications
    • Replicated synchronously to 3 AZ's
    • When you need real time, think Kinesis
    • Streaming data examples
      • Streaming processing frameworks
        • Spark
        • NiFi
      • Stock prices
      • Game data
      • Social network data
      • Geospatial data
      • Click stream data
    • There are four different kinds of Kinesis Streams
      • Data Streams: low latency streaming ingest at scale
      • Firehose Delivery stream: load streams into S3, Redshift, ElasticSearch and Splunk
      • Data Analytics: perform real time analytics on streams using SQL
      • Video Analytics: meant for streaming video in real time
    • Sample Architecture
      • [click streams, IoT devices, metrics and logs] --> Kinesis Streams --> Kinesis Analytics --> Kinesis Firehose --> [S3, Redshift]

    Kinesis Streams

    • Purpose: Stream big data in your system
    • Streams consist of shards and partitions
      • Shards are numbered (1, 2, 3, ..) and must be provisioned ahead of time
      • Shards define your capacity and ingestion rates
    • Characteristics of stream
      • Data retention is 24 hours by default with a max retention of 365 days
      • You can replay/reprocess this data within the retention period
      • Pub/sub architecture - multiple apps can consume the same stream
      • Once the data is inserted into Kinesis, it cannot be deleted (immutable)
      • Records can be up to 1mb in size
    • Capacity modes
      • Provisioned mode
        • You chose the number of shards provisioned
          • You can manually scale or use API
          • Each shard gets 1MB/s in (or 1000 records per second)
          • Each shard gets 2MB/s out (classic or enhanced fan out consumer)
          • Pay per shard per hour
      • On demand mode
        • No need to provision or manage the capacity
        • Default capacity provisioned (4Mb/s in or 4000 records per second)
        • Scales automatically based on observed throughput peak during the last 30 days
    • Limits to know
      • Producers
        • 1MB/s for 1000 messages/s at write per shard
          • If you exceed that limit, you will receive a "ProvisionedThroughputException"
      • Consumers
        • 2MB/s at read per shard across all consumers
        • 5 API calls per second per shard across all consumers
      • What does this mean? If you want to scale, you must increase shards

    Firehose

    • Designed to help capture, transform, and load streaming data into AWS data stores and analytics service
    • Pay for the data that goes through it
    • Functionality review
      • Near real time (see batching below)
        • Additionally you can only have 60 seconds layency minimum for non full batches
      • Data can be modified/transformed via Lambda
      • Takes care of the following tasks
        • Buffering
        • Compressing
        • Batching
      • Delivers to the following destinations
        • Primary
          • S3
          • Splunk
          • Redshift (via S3)
          • Elasticsearch
        • Secondary
          • Third party destinations (non-exclusive)
            • Datadog
            • mongoDB
            • Splunk
          • Other destinations that expose an HTTP endpoint
      • Backup data to S3
        • All source records
        • Transformation failiures
        • Delivery failures
    • Data formats
      • JSON
      • CSV
      • Parquet
    • Data Review
      • Conversions
        • CSV/JSON -> Parquet/ORC (only for S3)
        • Several templates are available
      • Transformations
        • Via AWS Lambda (ex: CSV -> JSON)
      • Compression
        • GZIP, ZIP, SNAPPY

    Analytics

    • Real-time data analytics service provided by Amazon Web Services (AWS) that allows you to process and analyze streaming data from various sources in real-time
    • Allows you to write SQL queries to analyze data streams from Amazon Kinesis Data Streams, Apache Kafka, or other streaming data sources
    • Inputs
      • Data streams
      • Firehose
      • S3 (for reference table)
    • Internal function
      • Allows SQL style queries to be executed
      • Can be combined with data from the reference table stored on S3
    • Outputs
      • Data streams
      • Firehose
    • Use cases
      • Streaming ETL
        • Reduce the size of your data set by selecting columns
        • Perform simple transformations
      • Continuous metric generation
        • Live leaderboards in video games
      • Responsive analytics
        • Look for certain criteria and build alerting
    • Features
      • Pay for resources consumed (but its not cheap)
      • Serverless and scales automatically
      • Use IAM permissions to access streaming sources and destinations
      • SQL or Flink to write computation
      • Schema discovery
      • Lambda can be used for pre-processing
    • Machine learning on Kinesis Data Analytics
      • Random Cut Forest
        • SQL function used for anomaly detection of numeric columns in a stream
        • Uses recent history to compute model (ie. the model is always changing
      • Hot spots
        • Locate information about relatively dense regions in your data

    Video Streams

    • Ingest video and audio encoded data from various devices and services
    • Output video data to ML or video processing services
    • Producers
      • One producer per video stream
      • Examples of producers
        • Security camaeras
        • body worn cameras
        • AWS Deep Lens
        • Smartphone camera
        • Radar data
    • Video playback capability
    • Consumers
      • Build your own (MXNet, Tensorflow)
      • AWS SageMaker
      • Amazon Rekognition Video
    • Data retention is from 1 hour -> 10 years
