What is it?
  • Fully managed deploy pipeline to staging or production environments
  • Can deploy on
    • EC2
      • Each EC2 instance must be running the CodeDeploy agent
    • On-premises
      • Each on-premise instance must be running the CodeDeploy agent
    • Lambda
    • ECS
  • Rapidly deploy new features
  • Update AWS Lambda functions versions
  • Avoid downtime during application deployment
  • Perform in place or blue/green deployments
  • Integrates with existing CI/CD tools such as Jenkins and CodePipeline
  • Integrates with existing config management tools such as Puppet, Chef, Ansible

How Does It Work?
  • Deployment Process
    • Each resource (EC2, on-prem, or Lambda) must contain the CodeDeploy agent
    • The agent continuously polls CodeDeploy for work to do
    • If work is found, CodeDeploy sends appspec.yml file
    • Application is pulled from GitHub or S3
    • EC2 will run the deployment instructions
    • CodeDeploy agent will report success/failure for specific instance
  • Additional Features and Considerations
    • EC2 instances are grouped by deployment group (for example, dev/test/prod)
    • CodeDeploy can be chained into CodePipeline and use artifacts from there
    • CodeDeploy can re-use existing setup tooks (Chef, Puppet), works with any application, and allows auto scaling integration
    • Note: Blue/Green deployments only work with EC2 instances as you cannot do this with on-prem servers
    • Code Deploy does not provision resources for you

Core Components

  • Application
    • Unique identifier for the application being deployed
    • Ensures the correct combination of revision, deployment config, and groups are referenced during deployment
  • Deployment Groups
    • Set of EC2 instances or Lambda functions where the new revision is deployed to
  • Deployment
    • Instance of a deployment group
    • Contains the revision, deployment config, and deployment group
    • Can be created manually or automatically
  • Deployment Configuration
    • Defines how a deployment is executed and rolled back
    • Can be created manually or automatically
  • AppSpec file
    • Contains instructions for how to deploy the application
    • Can be created manually or automatically
  • Revision
    • Version of the application to be deployed
    • Can be created manually or automatically

Deployment Types

  • In-place deployments
    • The app for each instance is stopped, the new version is deployed, and the app is started
    • You can use a load balancer so that each instance is de-registered during its deployment and then restored to service after deployment is complete
    • Only deployments that use the EC2/On-Premise compute platform can use in place deployments (no Lambda/ECS)
    • Strategy for taking instance offline
      • This strategy can be found further down the AWS Resource Groups configuration page
        • Built in options
          • CodeDeployDefault.OneAtATime
            • (Take one down, deploy, test, restore) x For each instance
          • CodeDeployDefault.HalfAtATime
            • (Half down, deploy, test, restore) x 2
          • CodeDeployDefault.AllAtOnce
            • Application will be offline for a bit. Good for development environment
        • Create deployment configuration
          • Define the minimum number of health hosts at a time
          • Can define that as a percentage or as a number
    • Blue/green deployments
      • See AWS/Beanstalk here
      • Deployment strategy that involves creating an identical production environment to accept deployments
        • The existing environment that is currently serving traffic is typically called the 'blue' environment
        • When deployment is initiated, an identical environment is created and typically called the 'green' environment
        • When a new version of the software is ready to be deployed, it is deployed to the idle "green" environment
        • Once the deployment is complete and the new version has been tested and verified, traffic is switched over from the "blue" environment to the "green" environment
        • This means that the "green" environment becomes live and starts serving user traffic, while the "blue" environment is destroyed
      • There are two environment configuration options available
        • Automatically copy Amazon EC2 Auto Scaling Group
          • Because this process creates new EC2 instances for each deployment, this is preferred
        • Manually provision instances
          • This is not the ideal solution due to the amount of work required
      • Load balancer
        • In this scenario, this is required
      • Optional wait time occurs for activities such as application testing and system verification
      • Instances in the replacement environment are registered with an ELB, causing traffic to be rerouted to them
      • Instances in the original environment are de-registered and can be terminated or used for debugging/rollback

    Appspec.yml

    • Responsible for describing where code should be installed and how to get it running
    • Can be created manually or automatically
    • Illustration (ExamPro)
    • Components
      • Choose OS version
      • Where to download the source
      • Apply permissions
      • Commands to stop the application (write bash files and point to the bash files in appspec.yml)
      • Commands to run before code is installed
      • Commands to run after code is installed
      • Commands to start the application

    Lifecycle Event Hooks

    • Allows you to view the events from a deploy to see if there is anything that failed and how long each event took.
    • Listed sequentially
    • Illustration (ExamPro)
    • The list includes any failures, which can be investigated further.. However some are cryptic and will require you to SSH into EC2 instance

    CodeDeploy Agent and ServiceRole

    • Need to install the CodeDeploy agent so the EC2 instance can report back to CodeDeploy
