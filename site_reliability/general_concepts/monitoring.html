    • Overview

    • Definitions
      • Monitoring
        • Collecting, processing, aggregating and displaying real time quantitative about a system
        • Examples include query counts and types, error counts and types, processing times, server lifetimes
        • It is typical to have one person who is best suited for this kind of development work
        • The goal is never to have "eyes on glass"
        • Its sometimes preferable to have simple and fast monitoring - avoiding "magic" systems
      • White box monitoring
        • Monitoring based on the metric exposed by the internals of the system
        • Includes logs, the underlying infrastructure, applications, and processes
        • The main advantage of white box monitoring is that it provides a complete view of the system being monitored, making it easier to identify and resolve issues
      • Black box monitoring
        • Focused on the external behavior and outputs of a system
        • Typically used to monitor the availability and response time of software systems, including web applications, APIs, and other services
        • The main advantage of black box monitoring is its simplicity
      • Gray box monitoring
        • Combines aspects of both white box and black box monitoring
        • Provides partial visibility into the internal workings of a system, without having complete access to the source code, configuration files, or other internal details
        • The main advantage of gray box monitoring is that it provides a good balance between detail and simplicity
      • Dashboard
        • Typically a web based application that provides a summary view of a services core metrics
        • May have filters and selectors but is basically prebuilt to expose a specific set of metrics
      • Alert
        • Notification intended to be read by a human
        • Typically pushed to an incident system, queue, email, etc
        • Alerts are typically triggered by a threshold being crossed
      • Root cause
        • A defect in a software or human system that, if fixed, instills confidence that the even will not happen again in the exact same way
        • Incidents may have multiple root causes

    Symptoms vs Causes

    • Monitoring should address two questions
      • What is broken
        • What is the symptom
        • What is the impact
      • Why is it broken
        • What are the causes
        • What is the root cause
      • Black box vs white box testing
        • Generally preferred to have heavy white box monitoring and modest black box monitoring
        • Black box monitoring is symptom oriented and represents active - not predicted - problems
        • White box monitoring can detect imminent, but yet to be realized, failures
